# yolovex

Interactive YOLO architecture explainer, inspired by [CNN Explainer](https://poloclub.github.io/cnn-explainer/). My goal was to dive into the new YOLO26 model from Ultralytics and understand the architecture. This project will let you trace how an image flows through the architecture of YOLO (currently YOLO26n) and to develop an intuition about what each block/layer/operation contributions to the final object detection. 

## Setup

```bash
uv sync
```

The first `yolovex` invocation will auto-download `yolo26n.pt` via Ultralytics — no manual weight download needed.

## Quick start

```bash
uv run yolovex predict          # head viz: boxes + per-scale heatmaps
open frontend/yolovex-l2.html   # L2 interactive page (uses committed data-l2.js)
```

## CLI

```bash
uv run yolovex layers                       # enriched block table
uv run yolovex describe 9                   # full info for one block
uv run yolovex show 5 --top 16              # render activations
uv run yolovex graph                        # paper-shaped SVG architecture graph
uv run yolovex build-assets                 # render block-level activations → frontend/data.js (L1)
uv run yolovex build-assets-l2              # render sub-module activations → frontend/data-l2.js (L2)
uv run yolovex predict                      # head viz — the "decision anchor"
uv run yolovex trace 200 400                # spatial-location trace through depth
uv run yolovex report                       # generates out/report.html
```

All accept `--image PATH`, `--imgsz N`, `--weights NAME`. Default image is `assets/sammit_lighthouse.jpg`.

## Key design decisions

- **Block-as-unit**, not layer or channel. The 24 yolo26 blocks (matching the YAML config) are what users click on. Inside a block, we pick channels by mean |activation| rank — channel order is a training artifact, not semantic.
- **Spatial location is the trace unit, not channel.** YOLO reshuffles channels at every 1×1 conv and Concat, so "channel 5 across layers" is meaningless. A spatial point on the input has a coherent identity at every depth (its receptive field grows). This is what `trace` follows.
- **Decision anchor first.** Mid-network feature maps mean nothing without a terminus. The detection head's per-scale class-score heatmaps are what turn "pretty pictures" into "the model decided X because of these activations."
- **Static HTML report as the bridge to a real frontend.** Forces us to make layout decisions properly (which sharpens the eventual frontend) without committing to a web stack. No JS, all images embedded base64, opens in any browser.
- **One inference per command** — no caching. ~1–2s on CPU at imgsz=640. Add a disk cache only if usage justifies it.
- **No tests yet.** Will add when something breaks. Trivial "model loads / PNG saves" tests rejected as theater.

## Fidelity ladder

The frontend is designed around a multi-level fidelity ladder so a learner can
choose how much detail they want without changing what model is being shown:

| Level | What's a node | Status |
|---|---|---|
| **L0** | One node per role (Backbone / Neck / Head) — the 30-second pitch | not started |
| **L1** | One node per top-level block (24 nodes total) | done — `yolovex.html` |
| **L2** | One node per **sub-module** inside each block. Conv → Conv2d/BN/SiLU; C3k2 → cv1/m.0/concat/cv2; SPPF → cv1/m.0/m.1/m.2/concat/cv2; C2PSA → cv1/m.0/cv2; Detect → P3/P4/P5 heads | done — `yolovex-l2.html` |
| **L3** | Drill into Bottleneck (cv1/cv2/residual) and PSABlock (attn/ffn) — the deepest unit users would meaningfully click on | not started |

### What L2 actually shows

- The 24 parent blocks become **labeled containers** that hold their sub-nodes
  in a nested layout. Each container's column position matches L1.
- Sub-modules are clickable; the side panel shows real input → output thumbnails
  (input is the previous sub-node's mean activation; output is the current one's),
  channel stack, and statistics — same UI as L1.
- For C3k2 / SPPF the internal layout uses a **staircase**: each tap that feeds
  the merge gets its own row, with input `i`'s left edge offset by
  `i × STAIR_FRAC × TILE_W` (default `STAIR_FRAC = 0.65`) from the merge's left
  edge. The merge tile spans the full width of all the taps it consumes
  (`TILE_W × (1 + STAIR_FRAC × (N-1))` for N inputs). Because every tap sits at
  a unique x position, the **tap → merge connection is just a straight vertical
  line at the tap's center x** — no curve, no detour, no bus routing.
  Sequential connections between consecutive taps are smooth right-angle
  beziers (out-right from one, then turn down into the top-mid of the next).
- All single-tile sub-nodes (Conv2d, BN, SiLU; cv1, cv2; PSABlock heads) are
  fixed at `TILE_W = 120` and **left-aligned at `innerLeft`** within their
  container. Combined with left-aligned containers (`x = COL_X[col]`), this
  makes inter-block forward arrows collapse to clean vertical lines —
  prev block's last sub and next block's first sub share an x.
- Atomic containers (Concat / Upsample at the block level) match
  `TILE_W + 2×pad = 152` so their centers line up with adjacent stack
  containers.
- The **Detect head card** (click the parent container) shows the input image
  with classic bbox + class/conf labels, plus a confidence-threshold slider
  that fades runners-up boxes in/out. The L2 builder captures up to ~50
  candidate boxes at conf=0.001 in `candidate_boxes` so the slider has
  something to reveal.
- A **flow overlay tile** floats over the top-right of the canvas: it always
  shows the active layer's mean thumbnail at a constant size, and switches to
  the annotated image when the Detect container is active. During play-flow
  (with a slow/medium/fast speed selector) it animates through every sub-node
  in dataflow order, ending on the annotated frame.

### Layout test canvas

`frontend/sppf-test.html` is a tiny standalone page that renders ONLY the
SPPF block (container + 6 sub-nodes + edges) with all layout constants and
bezier helpers inlined at the top of the file. It has no dependency on
`data.js`, `arch-l2.jsx`, or the rest of the L2 app — open it in a browser
and tweak the constants directly to iterate on the inside-the-block layout.
Use it whenever the staircase / fan-in geometry needs another tuning pass;
once the test canvas is happy, the same constants get ported back into
`layout-l2.jsx`.

### Why L2 needs its own data file

The L1 builder runs through Ultralytics' high-level predict path, which calls
`model.fuse()` internally — that folds `BatchNorm2d` weights into the preceding
`Conv2d` and the BN module's forward hook never fires. The L2 builder therefore
runs a **second forward pass** (`yolo.model(tensor)` directly, with manual
`LetterBox` preprocessing) which leaves BN as a separate module so its hook
fires and we get real BN activations. SiLU is shared across all Conv blocks in
yolo26's implementation, so we use the parent block's L1 output as the act
sub-node's output (mathematically identical — verified to 0.000000 max diff).

## Head structure (yolo26 specifics)

End-to-end head (NMS-free, dual one-to-many / one-to-one assignment, à la YOLOv10). Forward output is a tuple:
- `(B, max_det=300, 6)` — post-NMS boxes `(x1, y1, x2, y2, conf, cls)`
- `dict` with `one2many` and `one2one` branches; each contains `boxes (B,4,A)`, `scores (B,nc=80,A)`, `feats` (3 raw per-scale tensors at strides 8/16/32).

Anchors (`A=6300` at imgsz=640) decompose row-major across the per-scale feature maps. `head.split_scores_by_scale()` reshapes scores into `(nc, H, W)` per scale — that's the heatmap basis.

## Open questions / next iterations

- **L0 + L3** views, then a fidelity scrubber that switches between them on a single canvas (out of scope for the current iteration; L0 is trivial, L3 needs sub-sub-module hooks inside Bottleneck and PSABlock).
- **Per-page spacing controls** — bake the vertical / horizontal layout constants behind a small settings widget so a user can dial the diagram up to their preferred density.
- **Better arrow heads on L1** — port the L2 markers (notched-tail, `markerUnits="userSpaceOnUse"`) back to L1 so both pages match.
- **Bigger detection box / label rendering on L1** — the current L1 detection overlay uses thin strokes + tiny text; bump them once we standardize.
- Heatmap normalization currently min-max stretches per panel; the predict viz looks washed-out because background grid cells still get colored. Consider percentile clipping (e.g. `vmin=p50, vmax=p99`).
- Trace's bar chart is dominated by early layers (raw-pixel magnitudes). Per-layer normalization or log scale would surface the deeper-layer story better.
- Report layout is linear; a real FPN-shaped layout (rows = spatial scale, columns = depth) would mirror YOLO's actual structure but is significantly more layout work.
- Receptive field overlay on `trace`: show the actual receptive field around the marker, not just the center pixel.
- Ground-truth comparison: when bbox annotations exist, overlay them on the head viz to compare predicted vs actual.
